Hugo + PaperMod 派生主题：Windows 上部署到 GitHub Pages（HTTPS）完整指南#

适用场景：你从本仓库得到一个可运行的 Hugo 站点，希望在 Windows 上本地修改内容，并发布到 GitHub Pages（username.github.io）。

本指南默认使用 HTTPS + Token（PAT），不需要配置 SSH，能避开 Windows 中文用户名路径下 .ssh 相关问题。

你会得到什么#

  • 本地可预览：hugo server → http://localhost:1313/
  • 线上可访问：https://<username>.github.io/
  • 一套固定流程：写内容 → 生成 public/ → 推送到 Pages 仓库

1. 准备工作#

1.1 安装#

  • Git
  • Hugo（建议 extended 版）

安装完成后在 PowerShell 验证：

git --version
hugo version

1.2 建立 GitHub Pages 主站仓库#

  1. 登录 GitHub
  2. 新建仓库，仓库名 必须严格是：<username>.github.io
    • 例如用户名是 BXWang-IYU，仓库名就是：BXWang-IYU.github.io
  3. 建议选择 Public

仓库创建完成后：

  • 仓库地址：https://github.com/<username>/<username>.github.io
  • 站点地址：https://<username>.github.io/（通常显示为小写域名是正常的）

2. 获取并启动本仓库#

2.1 克隆代码（HTTPS）#

git clone https://github.com/xyming108/sulv-hugo-papermod.git
cd .\sulv-hugo-papermod

（可选）如果你想单独克隆一份 PaperMod 官方主题 作为参考/对比，也可以：

git clone https://github.com/adityatelange/hugo-PaperMod.git

注意：正常使用本仓库时，主题一般通过 git submodule update --init --recursive 拉到 themes/hugo-PaperMod/，不需要另外单独克隆一份。

如果本仓库使用了子模块（主题目录），执行：

git submodule update --init --recursive

2.2 本地预览#

hugo server

浏览器打开：http://localhost:1313/

3. 必改项：把站内链接从作者域名换成你的域名#

如果你发现点击菜单会跳转到作者示例站（例如 xyming108.github.io），根因通常是 config.yml 的 baseURL 还没改。

打开仓库根目录的 config.yml，修改为：

baseURL: https://<username>.github.io/

例子：

baseURL: https://bxwang-iyu.github.io/

修改后重新生成站点（见第 6 节），再推送 public/。

4. 常见兼容问题（Hugo 新版本）#

如果你使用的是较新的 Hugo（例如 v0.141+ / v0.158+ / v0.159+），可能会遇到一些“旧配置/旧模板”导致的报错。

4.0 先学会“按报错定位到具体文件”#

Hugo 的致命错误一般会给出 文件路径 + 行号，例如：

  • ... layouts/_default/baseof.html:5:8 ...
  • ... layouts/partials/head.html:63:86 ...

排查步骤建议固定为：

  1. 看最后一段 ERROR ... failed: "<路径>:<行>:<列>"，先打开这个文件

  2. 如果报错发生在 partial，继续打开报错里提到的那个 partials/**.html

  3. 同名文件如果同时存在：

    • layouts/**（你自己覆盖的模板）
    • themes/hugo-PaperMod/layouts/**（主题自带模板）

    优先检查 layouts/**，因为它会覆盖主题模板，很多兼容问题都来自这里

4.1 privacy.twitter.* / services.twitter.* 已移除#

报错类似：

  • privacy.twitter.enableDNT was deprecated ... removed. Use privacy.x.enableDNT instead.
  • services.twitter.disableInlineCSS ... removed. Use services.x.disableInlineCSS instead.

解决：把 config.yml 里的 privacy.twitter / services.twitter 改成 privacy.x / services.x。

4.2 site.Social / .Site.Author 字段取不到#

报错类似：

  • can't evaluate field Social in type interface {}
  • can't evaluate field Author in type page.Site

解决方向：

  • Social 信息建议放到 params.social（map）里读取
  • author 建议放到 params.author（map）里读取

本仓库如存在自定义覆盖模板（layouts/**），优先对照主题目录（themes/hugo-PaperMod/layouts/**）的最新实现调整。

4.3 resources.Concat expected slice of Resource objects#

如果你自定义的 layouts/partials/head.html 拼接 CSS 时引用了不存在的资源（例如某个 hljs 样式文件缺失），会导致 resources.Concat 类型错误。

典型报错形态：

  • error calling Concat: expected slice of Resource objects, received []interface {} instead

在本仓库的实际案例中，触发链路是：

  1. 报错指向 layouts/partials/head.html 的某一行（例如第 63 行）：
    • 它会把多个 CSS Resource 拼成一个文件：
      • slice $theme_vars $reset $common $hljs ... | resources.Concat ...
  2. 其中某个变量（常见是 $hljs）来自 resources.Get "css/..."，如果该文件不存在，就会变成空值/非 Resource
  3. resources.Concat 收到的切片里混入了非 Resource，导致类型错误

如何快速确认是否“缺资源”：

  • 去主题资源目录确认文件是否存在，例如：themes/hugo-PaperMod/assets/css/**
  • 或临时把相关资源从 slice(...) 中移除/替换为 FromString 生成的空白资源来验证

一个最简单的绕过方法是在 config.yml 中关闭 HLJS：

params:
  assets:
    disableHLJS: true

4.4 本仓库实际遇到的 3 个“致命报错”怎么修（可照抄）#

A) resources.Concat ... expected slice of Resource objects#

现象：hugo server 直接中断，报错中包含 resources.Concat，并指向 layouts/partials/head.html。

处理（最省事）：在 config.yml 增加：

params:
  assets:
    disableHLJS: true

B) twitter_cards.html/opengraph.html 报 site.Social#

现象：报错中包含：

  • partials/templates/opengraph.html ... <site>: can't evaluate field Social in type interface {}
  • 或 partials/templates/twitter_cards.html ... <site>: can't evaluate field Social in type interface {}

原因：旧模板用 site.Social.*，新 Hugo/主题实现改为从 site.Params.social（map）读取。

处理方式：对照 themes/hugo-PaperMod/layouts/partials/templates/*.html 的最新写法，把 layouts/partials/templates/*.html 里对应字段替换为 site.Params.social。

C) rss.xml 报 .Site.Author.email#

现象：报错中包含：

  • layouts/_default/rss.xml ... <.Site.Author.email>: can't evaluate field Author in type page.Site

原因：旧模板用 .Site.Author.*，但新版本 Hugo 不再提供该字段。

处理方式：改为从 site.Params.author（map）读取（参考主题 themes/hugo-PaperMod/layouts/_default/rss.xml 的实现）。

5. 添加你自己的内容#

5.1 写文章#

文章一般放在：

  • content/posts/

推荐用命令生成（会带 front matter）：

cd <项目根目录>
hugo new posts/my-first-post.md

然后编辑生成的 content/posts/my-first-post.md。

5.2 修改页面#

常见页面文件（按仓库实际为准）：

  • content/about.md（关于）
  • content/links.md（友链）
  • content/archives.md（归档）
  • content/search.md（搜索）

5.3 放图片/静态资源#

放到：

  • static/

例如放在 static/img/a.png，文章中引用：

![](/img/a.png)

6. 生成 public/（发布产物）#

在项目根目录运行：

hugo

生成后的静态网站文件在 public/。

7. 发布到 GitHub Pages（教程流：推送 public/）#

这种方式的要点：Pages 仓库里放的是 public/ 的内容（纯静态文件），不是 Hugo 源码工程。

7.1 初始化 public/ 为一个独立 Git 仓库（只做一次）#

cd .\public
git init
git branch -M main
git remote add origin https://github.com/<username>/<username>.github.io.git

7.2 首次提交并推送#

git add -A
git commit -m "deploy"
git push -u origin main

如果提示没有设置提交身份（仅影响 commit 显示，不是登录）：

git config --global user.name "<你的名字>"
git config --global user.email "<你的邮箱>"

7.3 HTTPS 登录方式（重要）#

当 git push 要求输入用户名/密码时：

  • Username：你的 GitHub 用户名
  • Password：粘贴你的 Token（PAT）（不是 GitHub 登录密码）

Token 获取路径：

GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate

常用勾选：repo

7.4 更新发布（每次改内容之后）#

每次你修改内容后执行：

cd <项目根目录>
hugo
cd .\public
git add -A
git commit -m "update"
git push

8. Pages 设置检查（网页端）#

进入仓库 https://github.com/<username>/<username>.github.io：

Settings → Pages：

  • Build and deployment：Deploy from a branch
  • Branch：main
  • Folder：/(root)

9. 常见报错速查#

9.1 error: src refspec main does not match any#

原因：你本地没有 main 分支（常见是当前分支还是 master）。

解决：

git branch -M main
git push -u origin main

9.2 线上点击链接仍跳到别人域名#

优先检查：

  • config.yml 的 baseURL 是否已改成你的域名
  • 是否重新运行了 hugo 生成新的 public/
  • 是否把新的 public/ 成功 git push 到 username.github.io 仓库
  • 浏览器是否缓存：尝试 Ctrl + F5 或无痕窗口

10. 推荐：把源码与产物分开（进阶）#

本指南采用“推送 public/”的教程流，简单但维护成本略高（源码与产物分离）。

更规范的做法是：

  • Hugo 源码放一个仓库
  • 用 GitHub Actions 构建
  • 输出自动部署到 Pages

需要的话可以在此基础上再升级。
